# Telemetry recording function

## Summary

A Cloud Function that records CLI installations: it creates or updates the repository and
skills, increments installation counts, and marks the affected skill(s) as awaiting
validation.

## Acceptance criteria

### AC-1: Installation recorded

**Given** a telemetry request for an installed repository and skill(s)
**When** the function processes it
**Then** the repository/skills are created or updated, install counts incremented, and affected skills marked pending
**And** timestamps are updated.

### AC-2: Malformed telemetry rejected

**Given** a telemetry payload missing required fields
**When** the function is invoked
**Then** it returns a client error and records nothing
**And** existing counts are unchanged.

### AC-3: Boundary — idempotent / consistent under retries

**Given** the same installation telemetry is delivered more than once due to retries
**When** the function processes the duplicates
**Then** counts increment according to the documented rule without corrupting aggregates
**And** the stored state remains consistent.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → installation recorded and counts incremented
2. **Critical negative** → malformed telemetry rejected
3. **Permission boundary** → retries handled consistently

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
