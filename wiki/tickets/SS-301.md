# Repository scan for SKILL.md up to depth 3

## Summary

Using the GitHub API, scan a target repository for skills defined by a `SKILL.md` file
located at most three levels deep (root, `/skills/<name>/`, `/skills/<group>/<name>/`),
ignoring anything nested deeper.

## Acceptance criteria

### AC-1: Skills found at allowed depths

**Given** a repository with `SKILL.md` files at the root, at depth 2, and at depth 3
**When** the CLI scans the repository
**Then** all three skills are discovered and reported
**And** each is identified by its containing directory.

### AC-2: Deeper nesting ignored / repo not found

**Given** a `SKILL.md` located deeper than three levels, or a nonexistent repository URL
**When** the CLI scans
**Then** the too-deep skill is not listed, and an invalid URL yields a clear "repository not found/accessible" error
**And** the command exits non-zero on the invalid URL.

### AC-3: Boundary — GitHub API rate limit / auth

**Given** the GitHub API returns a rate-limit or auth error during scan
**When** the CLI scans
**Then** it surfaces a clear message about the limit/authentication and how to proceed
**And** it does not partially report an incomplete skill list as complete.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → skills discovered at depths 1-3
2. **Critical negative** → deeper nesting ignored; invalid repo errors
3. **Permission boundary** → rate-limit/auth errors handled clearly
