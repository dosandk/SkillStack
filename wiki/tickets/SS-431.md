# Profile page with GitHub information

## Summary

Give a logged-in user a profile page that displays their GitHub information (such as
username, avatar, and profile details).

## Acceptance criteria

### AC-1: Profile shows GitHub info

**Given** a logged-in user
**When** they open their profile page
**Then** their GitHub information is displayed
**And** it reflects the account used to sign in.

### AC-2: GitHub info fetch fails

**Given** GitHub profile data cannot be retrieved
**When** the profile page loads
**Then** a graceful error/placeholder is shown
**And** the page does not crash.

### AC-3: Boundary — profile requires authentication

**Given** an unauthenticated visitor
**When** they navigate to the profile route
**Then** they are redirected to sign in
**And** no profile data is exposed.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → GitHub info displayed on profile
2. **Critical negative** → fetch failure handled gracefully
3. **Permission boundary** → profile requires authentication

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
