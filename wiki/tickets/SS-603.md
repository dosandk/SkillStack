# On-demand validation for owners

## Summary

Expose validation as an on-demand operation that a logged-in owner can trigger for their
own skill or entire repository, fetching the latest GitHub version each time.

## Acceptance criteria

### AC-1: Owner triggers on-demand validation

**Given** a logged-in owner of a repository/skill
**When** they request validation on demand
**Then** validation runs against the latest GitHub version and stores the result
**And** the updated status is returned.

### AC-2: Validation of nonexistent/removed content

**Given** an entry whose GitHub source no longer exists
**When** on-demand validation runs
**Then** it reports a failure that the source could not be fetched
**And** does not mark the entry approved.

### AC-3: Boundary — only owner may trigger

**Given** a user who is not the owner
**When** they attempt to trigger validation for the entry
**Then** the request is rejected as unauthorized
**And** no validation is run.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → owner runs validation, result stored
2. **Critical negative** → missing source fails validation
3. **Permission boundary** → only owner may trigger
