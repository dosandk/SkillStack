# Validate button triggers on-demand validation

## Summary

Give the content owner a Validate button on their repository or individual skill that
triggers on-demand validation, which always fetches the latest version from GitHub.

## Acceptance criteria

### AC-1: Owner triggers validation

**Given** a logged-in owner viewing their pending skill/repository
**When** they click Validate
**Then** on-demand validation is started against the latest GitHub version
**And** the UI shows an in-progress state until results return.

### AC-2: Validation service failure

**Given** the validation service errors or times out
**When** the owner triggers validation
**Then** the UI shows a failure message and leaves the prior status unchanged
**And** the owner can retry.

### AC-3: Boundary — non-owner cannot validate

**Given** a logged-in user who is not the owner
**When** they view someone else's entry
**Then** the Validate action is unavailable to them
**And** a direct validation request for that entry is rejected.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → owner starts validation, sees in-progress state
2. **Critical negative** → validation failure surfaced, status preserved
3. **Permission boundary** → non-owner cannot validate

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
