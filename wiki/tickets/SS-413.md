# Restrict upload to authenticated users

## Summary

Enforce that only authenticated users can perform uploads, both in the UI and at the
backend, so anonymous submissions are rejected.

## Acceptance criteria

### AC-1: Authenticated upload allowed

**Given** an authenticated user
**When** they submit an upload
**Then** the request is authorized and processed
**And** the entry is attributed to them.

### AC-2: Unauthenticated request rejected server-side

**Given** an upload request without a valid session/token
**When** it reaches the backend
**Then** it is rejected with an unauthorized error
**And** no entry is created.

### AC-3: Boundary — expired session at submit time

**Given** a user whose session expired between page load and submit
**When** they submit the upload
**Then** the backend rejects it as unauthorized and the UI prompts re-auth
**And** the submission is not silently dropped or accepted.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → authenticated upload authorized
2. **Critical negative** → unauthenticated request rejected server-side
3. **Permission boundary** → expired session rejected with re-auth prompt

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
