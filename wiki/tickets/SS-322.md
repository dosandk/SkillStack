# Commit-hash update detection

## Summary

Compare the repository's latest commit hash against the hash stored in Firestore to
detect whether the repository owner has updated it since it was last recorded.

## Acceptance criteria

### AC-1: No change detected

**Given** the latest commit hash equals the stored hash
**When** the CLI compares them
**Then** it treats the stored data as current
**And** proceeds without prompting about updates.

### AC-2: Change on an unvalidated stored version

**Given** the latest commit differs from a stored version that has not been validated
**When** the CLI compares them
**Then** it installs the selected skill(s) as in the first-install flow
**And** does not prompt for a version choice.

### AC-3: Boundary — change on a validated stored version

**Given** the latest commit differs from a stored version that has already been validated
**When** the CLI compares them
**Then** it flags that changes occurred and defers to the version-choice flow (SS-323)
**And** does not auto-install the updated version.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → matching hashes treated as current
2. **Critical negative** → change on unvalidated version installs directly
3. **Permission boundary** → change on validated version requires user choice

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
