# Reflect updated validation status in UI

## Summary

After validation completes, update the entry's displayed status (approved/failed) in the
UI and in listings so users see the current result.

## Acceptance criteria

### AC-1: Status updates after validation

**Given** a validation that just completed for an owner's entry
**When** results are stored
**Then** the entry's status badge updates to the new value in the detail view
**And** listings reflect the same status.

### AC-2: Stale status not shown

**Given** validation changed an entry from pending to failed
**When** the user revisits the entry
**Then** the failed status is shown rather than the stale pending value
**And** the change is consistent across views.

### AC-3: Boundary — repo status derived from skills

**Given** a repository whose skills have mixed statuses
**When** the repository status is displayed
**Then** the calculated repo-level status matches the documented aggregation rule
**And** it is consistent with the per-skill statuses shown.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → status updates after validation
2. **Critical negative** → stale status not shown
3. **Permission boundary** → repo status derived from skills consistently

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
