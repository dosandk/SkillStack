# Create pending-status registry entry

## Summary

On upload, create the registry entry — a single skill, or a repository with its list of
skills — in Firestore with `pending` validation status, and show it in the UI as
awaiting validation.

## Acceptance criteria

### AC-1: Repository entry created as pending

**Given** a valid repository URL with multiple skills
**When** the upload is processed
**Then** a repository entry plus its skills subcollection are created with `pending` status
**And** the entry appears in the UI marked pending.

### AC-2: Duplicate upload handled

**Given** a repository already present in the registry
**When** the same URL is uploaded again
**Then** the system reuses/updates the existing entry rather than creating a duplicate
**And** the user is informed it already exists.

### AC-3: Boundary — ownership recorded to uploader

**Given** a logged-in user uploads a repository
**When** the entry is created
**Then** the uploading user is recorded as the owner
**And** only they can later trigger owner-only actions on it.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → repository + skills created pending
2. **Critical negative** → duplicate upload deduplicated
3. **Permission boundary** → ownership recorded to uploader
