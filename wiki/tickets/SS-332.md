# Mark installed repo/skills as awaiting validation

## Summary

The telemetry request marks the repository with the specified set of skills (or a single
skill) as awaiting later validation, so the daily job and owners know it needs checking.

## Acceptance criteria

### AC-1: New repo recorded as awaiting validation

**Given** a repository not previously in Firestore
**When** telemetry is recorded after install
**Then** the repository and its installed skill(s) are stored with `pending` validation status
**And** timestamps and installation count are initialized.

### AC-2: Existing validated repo re-flagged on updated install

**Given** a previously validated repository installed at an updated commit
**When** telemetry is recorded
**Then** the affected skill(s) are marked as awaiting validation again
**And** the installation count is incremented.

### AC-3: Boundary — only installed skills are flagged

**Given** a repository containing skills that were not installed in this run
**When** telemetry is recorded
**Then** only the installed skill(s) are marked awaiting validation
**And** untouched skills retain their prior status.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → new repo stored as pending
2. **Critical negative** → updated validated repo re-flagged pending
3. **Permission boundary** → only installed skills flagged
