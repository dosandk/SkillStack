# Search repositories

## Summary

Allow users to search for entire repositories (not just individual skills) and view a
repository with its list of validated skills.

## Acceptance criteria

### AC-1: Repository search returns repos

**Given** repositories that contain validated skills
**When** a visitor searches by keyword
**Then** matching repositories are listed with their validated skills
**And** the repository can be opened to see its skill list and metadata.

### AC-2: Repository with no validated skills

**Given** a repository whose skills are all pending/failed
**When** it would match a query
**Then** it is not surfaced as validated public content
**And** the search returns no misleading repository entry.

### AC-3: Boundary — repo vs skill result types are distinct

**Given** a query matching both a repository name and a skill name
**When** results render
**Then** repository and skill results are clearly distinguished
**And** each links to the correct detail view.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → repositories with validated skills returned
2. **Critical negative** → repo without validated skills excluded
3. **Permission boundary** → repo and skill result types distinguished

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
