# Calculated repo validation status & install count

## Summary

Compute and store the repository-level validation status and total installation count as
derived fields aggregated from the `skills` subcollection, keeping them consistent as
skills change.

## Acceptance criteria

### AC-1: Aggregates reflect skills

**Given** a repository with several skills of differing statuses and install counts
**When** aggregation runs
**Then** the repo validation status and total install count reflect the documented rules over its skills
**And** the stored aggregate values match the underlying skills.

### AC-2: Aggregate updates on skill change

**Given** a skill's status or install count changes
**When** the change is committed
**Then** the repository-level aggregates are recomputed
**And** they do not go stale.

### AC-3: Boundary — empty skills subcollection

**Given** a repository with no skills
**When** aggregates are computed
**Then** a defined default status and zero install count are stored
**And** no divide-by-zero or undefined state occurs.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → aggregates reflect skills
2. **Critical negative** → aggregates update on skill change
3. **Permission boundary** → empty subcollection handled with defaults

- Concurrent access scenarios (if applicable)
