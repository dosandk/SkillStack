# GitHub OAuth login flow

## Summary

Allow a user to sign in with their GitHub account via OAuth so the app can establish an
authenticated identity used for uploads, validation, and profile access.

## Acceptance criteria

### AC-1: Successful sign-in

**Given** an unauthenticated visitor on the app
**When** they click "Sign in with GitHub" and approve the OAuth consent
**Then** they are redirected back authenticated and their GitHub identity is available to the app
**And** a session/token is established for subsequent requests.

### AC-2: Denied consent

**Given** a visitor who starts the GitHub OAuth flow
**When** they cancel or deny the consent screen
**Then** they return to the app still unauthenticated with a non-blocking message
**And** no session is created.

### AC-3: Boundary — protected action without login

**Given** an unauthenticated user
**When** they attempt to open an authenticated-only route (e.g. upload)
**Then** they are redirected to sign in
**And** the protected action is not performed.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → successful GitHub sign-in
2. **Critical negative** → denied/cancelled consent leaves user unauthenticated
3. **Permission boundary** → protected route redirects unauthenticated users

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
