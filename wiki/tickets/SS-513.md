# GitHub API enrichment (stars & live metadata)

## Summary

Provide a backend path that queries the GitHub API to fetch up-to-date repository
metadata (such as stars) and merges it with stored Firestore data for display in the UI.

## Acceptance criteria

### AC-1: Live metadata merged

**Given** a repository present in Firestore
**When** the enrichment path is called
**Then** stored data is returned merged with live GitHub metadata such as stars
**And** the merged result identifies stored vs. live fields consistently.

### AC-2: GitHub API failure degrades

**Given** the GitHub API errors or is rate-limited
**When** enrichment is requested
**Then** stored data is still returned and live fields are omitted/marked stale
**And** the call does not fail hard.

### AC-3: Boundary — caching/rate-limit respected

**Given** repeated enrichment requests within a short window
**When** they are served
**Then** results are cached/throttled to respect GitHub rate limits
**And** stored data remains authoritative for validation/commit fields.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → live metadata merged with stored data
2. **Critical negative** → GitHub failure degrades gracefully
3. **Permission boundary** → caching/rate-limit respected

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
