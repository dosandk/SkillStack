# URL upload input & submission

## Summary

Provide a logged-in user with a URL input field to submit a GitHub repository or skill
for inclusion in the registry.

## Acceptance criteria

### AC-1: Valid URL submitted

**Given** a logged-in user on the upload view
**When** they enter a valid GitHub repository/skill URL and submit
**Then** the submission is accepted and an entry-creation flow begins
**And** the user sees confirmation that upload is in progress.

### AC-2: Invalid URL rejected

**Given** a logged-in user
**When** they submit a malformed or non-GitHub URL
**Then** an inline validation error is shown
**And** no entry is created.

### AC-3: Boundary — unauthenticated cannot upload

**Given** an unauthenticated visitor
**When** they attempt to reach the upload input
**Then** they are prompted to sign in
**And** the upload field is not usable until authenticated.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → valid URL accepted
2. **Critical negative** → invalid URL rejected inline
3. **Permission boundary** → unauthenticated users cannot upload

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
