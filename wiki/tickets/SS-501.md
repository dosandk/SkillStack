# Repository document schema

## Summary

Define the Firestore repository document in the single top-level collection: owner
reference, repository reference, GitHub commit hash, short README-derived description,
calculated repo-level validation status, calculated installation count, and
created/updated timestamps. Skill files are never stored — only the commit hash.

## Acceptance criteria

### AC-1: Repository document created with required fields

**Given** a new repository is registered
**When** its document is written
**Then** it contains owner, repository reference, commit hash, description, calculated validation status, calculated install count, and timestamps
**And** no raw skill file content is stored.

### AC-2: Invalid document rejected

**Given** a write missing required fields or with wrong types
**When** it is attempted
**Then** the write is rejected by schema/validation
**And** no partial document is persisted.

### AC-3: Boundary — timestamps managed by the system

**Given** a client attempts to set created/updated timestamps directly
**When** the write is processed
**Then** the server-managed timestamps take precedence
**And** client-supplied timestamp overrides are ignored.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → document created with all required fields
2. **Critical negative** → invalid document rejected
3. **Permission boundary** → timestamps system-managed
