# Client-side auth state & session guard

## Summary

Maintain authenticated state on the client and guard authenticated-only routes/actions,
exposing the current user to components and clearing state on sign-out.

## Acceptance criteria

### AC-1: Auth state persists across reloads

**Given** a signed-in user
**When** they reload the app
**Then** their authenticated state is restored without re-prompting for login
**And** the current user's GitHub info is available to components.

### AC-2: Expired session is handled

**Given** a user whose session/token has expired
**When** they trigger a request requiring auth
**Then** they are treated as unauthenticated and prompted to sign in again
**And** stale user state is cleared from the client.

### AC-3: Boundary — sign-out revokes access

**Given** a signed-in user
**When** they sign out
**Then** authenticated-only routes become inaccessible immediately
**And** the session is cleared.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → auth state restored on reload
2. **Critical negative** → expired session prompts re-auth
3. **Permission boundary** → sign-out revokes protected access

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
