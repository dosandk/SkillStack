# Version choice for validated repos with changes

## Summary

When a validated stored version differs from the latest commit, let the user choose
which version to install: the validated stored version or the updated latest version.

## Acceptance criteria

### AC-1: User picks the validated version

**Given** a validated stored version differs from the latest commit
**When** the user is prompted and selects the validated version
**Then** the CLI installs the validated stored version
**And** no telemetry marking it as awaiting validation is triggered for a new version.

### AC-2: User picks the updated version

**Given** the same difference is detected
**When** the user selects the updated latest version
**Then** the CLI installs the latest version
**And** triggers telemetry recording a new installation awaiting future validation.

### AC-3: Boundary — user cancels the choice

**Given** the version-choice prompt is shown
**When** the user cancels
**Then** nothing is installed
**And** the command exits without recording an installation.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → validated version installed on request
2. **Critical negative** → updated version installed + telemetry queued
3. **Permission boundary** → cancel installs nothing

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
