# Firestore security rules for repos & skills

## Summary

Author Firestore security rules so public reads are limited to validated content, owners
can manage their own repositories/skills, and derived fields (validation status, install
counts) are writable only by trusted functions.

## Acceptance criteria

### AC-1: Owner can manage own content

**Given** an authenticated owner
**When** they modify their own repository/skill within allowed fields
**Then** the write succeeds
**And** ownership is enforced by the rules.

### AC-2: Non-owner write rejected

**Given** an authenticated user who is not the owner
**When** they attempt to modify another user's repository/skill
**Then** the write is denied by the rules
**And** the data is unchanged.

### AC-3: Boundary — derived fields function-only

**Given** any client (owner or not)
**When** it attempts to write calculated validation status or install counts directly
**Then** the write is denied and only trusted functions may set them
**And** public reads remain limited to validated content.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → owner manages own content
2. **Critical negative** → non-owner write denied
3. **Permission boundary** → derived fields writable only by functions

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
