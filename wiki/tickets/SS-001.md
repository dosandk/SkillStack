# Monorepo scaffold + TypeScript path aliases

## Summary

Restructure the repository into a single-package monorepo (`client`, `functions`,
`shared`, `wiki`) plus a standalone `cli` package, and wire cross-folder imports
through TypeScript path aliases (`@shared`, `@db`, `@eleks-ui/*`) so both the front end
and Cloud Functions share a single source of truth.

## Acceptance criteria

### AC-1: Aliased imports resolve across folders

**Given** the monorepo with `client/` and `functions/` present
**When** a developer imports a type from `@shared` in both the client and a function
**Then** `npm run typecheck` passes with no unresolved-path errors
**And** the compiled output resolves the alias to the `shared/` source.

### AC-2: Disallowed cross-boundary import fails fast

**Given** a module attempts to import from a folder not exposed via an alias
**When** `npm run typecheck` runs
**Then** the build fails with a clear module-resolution error
**And** no partial build artifact is emitted.

### AC-3: Boundary — CLI stays an independent package

**Given** the standalone `cli` package
**When** its dependencies are installed and it is built in isolation
**Then** it builds without requiring the root package
**And** it does not import from `client/` or `functions/` internals.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → aliased import + typecheck passes
2. **Critical negative** → disallowed import fails the build
3. **Permission boundary** → CLI builds independently of the root package

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
