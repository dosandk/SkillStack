# LLM validation core (security + conventions)

## Summary

Implement the core validation that uses an LLM (e.g. Anthropic SDK) to check a skill's
latest GitHub version for security issues and adherence to documented conventions,
producing a critical/pass verdict.

## Acceptance criteria

### AC-1: Clean skill passes

**Given** a skill with no security or convention violations
**When** validation runs
**Then** the LLM check returns no critical issues and the skill is marked approved
**And** the result is persisted with the validated commit hash.

### AC-2: Critical issue fails validation

**Given** a skill containing a security or convention violation
**When** validation runs
**Then** the skill is marked failed with the critical issue captured
**And** it is not marked approved.

### AC-3: Boundary — validation always uses latest GitHub version

**Given** a stored commit hash older than the current GitHub HEAD
**When** validation runs
**Then** it fetches and evaluates the latest GitHub version
**And** records the commit hash it actually validated.

## E2E test generation guidance

The three acceptance criteria above map directly to the minimum E2E coverage:

1. **Golden path** → clean skill approved
2. **Critical negative** → violation marks skill failed
3. **Permission boundary** → validation always uses latest GitHub version

Additional E2E tests should be added for:

- Data edge cases mentioned in acceptance criteria
- Concurrent access scenarios (if applicable)
- Cross-feature interactions (if this ticket's feature depends on others)
