# Telemetry recording function

## Summary

A Cloud Function that records CLI installations: it creates or updates the repository and
skills, increments installation counts, and marks the affected skill(s) as awaiting
validation.

This function is called anonymously (no logged-in user — the CLI has no auth context)
and writes install counts, not an owner. `SS-412` (the web upload story's create-entry
function) upserts the same kind of repository/skills document but requires an
authenticated caller and writes an `owner` field instead. The two remain separate Cloud
Functions with different auth requirements, but should share the same underlying
"upsert repository/skills document, defaulting new ones to `pending`" helper rather than
each reimplementing that create-vs-reuse logic independently.

## Unit test requirements

- Payload-validation logic: given a telemetry request body, determines whether
  required fields are present before any Firestore write is attempted.
- Count-increment/idempotency logic: given a stored install count and a request that
  may be a retried duplicate, computes the correct resulting count without
  double-counting.

## Integration test requirements

- Invoke the function against the Firestore emulator with a valid telemetry payload —
  assert the repository/skills document is created or updated, install count
  incremented, affected skills marked pending, and timestamps updated.
- Invoke the function against the emulator with a payload missing required fields —
  assert it returns a client error and the emulator document/counts are unchanged.
- Invoke the function against the emulator twice with the same installation telemetry
  (simulating a retry) — assert the stored count reflects the documented idempotency
  rule (not double-incremented) and the stored state remains consistent.
