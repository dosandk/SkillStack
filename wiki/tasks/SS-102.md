# `firestore.rules` deny-all

## Summary

Add `firestore.rules` (it doesn't exist today — `firebase.json`'s `firestore` key only
references `firestore.indexes.json`) denying all client reads and writes on every
collection, and wire it into `firebase.json` (`"rules": "firestore.rules"`). Per [AD-1](../architecture-invariants.md#ad-1)/
[AD-3](../architecture-invariants.md#ad-3): only the Cloud Functions admin SDK ever touches Firestore, so no owner-based rule
logic is needed — `allow read, write: if false` for everything ([NFR1](../requirements.md#nfr1)).

## Unit test requirements

N/A — rules are configuration, not unit-testable code. Verified entirely through the
integration tests below.

## Integration test requirements

- A rules-emulator test (`@firebase/rules-unit-testing`) asserts an unauthenticated
  client read on any collection is denied.
- A rules-emulator test asserts an authenticated client write is denied (no owner
  exception exists, confirming [AD-3](../architecture-invariants.md#ad-3)'s deny-all is real, not accidentally scoped).
- A rules-emulator test confirms the admin SDK (server context) bypasses the rules as
  expected — Cloud Functions still work end to end.

## Quality gates

- [ ] Rules file contains no owner-based exception for any collection
- [ ] Integration tests cover every case listed above
- [ ] `firebase.json` references the rules file
- [ ] Story's task table updated (status reflects this task)
