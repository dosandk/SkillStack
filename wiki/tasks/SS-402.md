# On-demand + scheduled validation triggers

## Summary

Two thin triggers wrapping SS-401's validation service, per [FR-BE4](../requirements.md#fr-be4): an owner-scoped
`onRequest` endpoint ([AD-9](../architecture-invariants.md#ad-9) — verifies the caller's ID token and that they own the
repo/skill) for the "Validate" button, and a daily scheduled Cloud Function that runs
the same service against every skill whose status is not yet `validated`/`failed`, or
whose commit hash has changed since its last check.

## Unit test requirements

- The on-demand endpoint rejects a caller who isn't the repo's owner.
- The on-demand endpoint, given a valid owner token, invokes the validation service for
  the specified skill (or every skill in the repo, if none specified).
- The scheduled function selects exactly the skills that are pending or stale
  (commit hash changed), not ones already validated at the current commit.

## Integration test requirements

- The on-demand endpoint, called against the Firestore emulator with a valid owner
  token, updates the expected skill's `findings` and status.
- The scheduled function, run against the emulator with a mix of pending/validated/
  stale skills, only updates the ones it should.
