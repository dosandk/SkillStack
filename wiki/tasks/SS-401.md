# Validation service

## Summary

The core validation logic, called by both triggers in SS-402: for a given skill, fetch
its latest content from GitHub (always fresh, never cached — [FR-UI8](../architecture.md#fr-ui8)), call the
Anthropic SDK to check it for security/convention issues (critical) and best-practice
gaps (non-critical), and write the result as a `findings` array
(`{ severity: 'critical' | 'recommendation', message: string }[]`) on that skill's
Firestore doc. Then call `repositories-store.ts`'s `recalculateValidationStatus(repoId)`
(AD-13) synchronously, in the same invocation, to roll that up onto the repository
doc: `validated` only if every skill has zero critical findings, otherwise `failed`;
`pending` skills not yet checked don't affect the calculation. This function touches
validation status only, never install count — the counterpart to SS-504's
`recalculateInstallCount` call, per AD-11/AD-13's one-field-per-helper split.

## Unit test requirements

- Given a mocked Anthropic response with at least one critical finding, the skill's
  calculated status becomes `failed` and both severities are stored.
- Given a mocked response with only recommendations, the skill's status becomes
  `validated`.
- `recalculateValidationStatus` is called after the skill write, and the
  repository-level status it produces reflects the worst status among its skills.
- The GitHub content fetch always requests the current default branch/latest commit,
  never a cached one.
- This function never writes a skill's or repository's `installCount` field — only
  `validationStatus`/`findings`.

## Integration test requirements

- Running the service against the Firestore emulator with a stubbed Anthropic client
  and a stubbed GitHub fetch persists the expected `findings` array and calculated
  statuses at both skill and repository level.
