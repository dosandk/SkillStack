# Mark installed repo/skills as awaiting validation

## Summary

The telemetry request marks the repository with the specified set of skills (or a single
skill) as awaiting later validation, so the daily job and owners know it needs checking.

## Unit test requirements

- Status-transition logic: given a current stored validation status (none/pending/
  approved/failed) and an incoming install event, computes the resulting status per
  skill, as a pure function with no Firestore access.
- Skill-selection logic: given the installed-skill subset from a telemetry payload,
  determines which skills within a repo document should be updated versus left
  untouched.

## Integration test requirements

- Invoke the telemetry-recording path against the Firestore emulator for a repository
  with no existing document — assert the repository and skill(s) are written with
  `pending` status and timestamps/installation count are initialized.
- Seed the emulator with a previously validated repository, then invoke recording for
  an install at an updated commit — assert the affected skill(s) are re-flagged
  pending and the installation count is incremented.
- Seed the emulator with a repo containing skills beyond those installed in this run —
  assert only the installed skill(s) are updated to pending and untouched skills'
  stored status is unchanged.
