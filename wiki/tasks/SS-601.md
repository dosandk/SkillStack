# Repo-status response contract

## Summary

`scanRepository` (SS-301) already returns `{ known: true, commitHash, statuses }` for a
tracked repo — this task locks down and documents that exact response shape as a
concrete sample JSON in this file (AD-5: no shared types package between `cli/` and
`functions/`, so the contract is documented by example instead), since the CLI's reinstall
path (SS-602/SS-603) depends on it precisely.

**Sample contract:**
```json
{
  "known": true,
  "commitHash": "a1b2c3d",
  "repoValidationStatus": "validated",
  "skills": [
    { "name": "frontend-design", "path": "skills/frontend-design", "validationStatus": "validated" }
  ]
}
```

## Unit test requirements

- The response includes `commitHash` and, per skill, its own `validationStatus` — not
  just a single repo-level status.
- The response for an unknown repo is `{ "known": false }` with no other fields assumed
  present (matches SS-301, re-verified here from the CLI-contract angle).

## Integration test requirements

- Calling `scanRepository` against the Firestore emulator for a previously-uploaded repo
  returns a response matching the sample contract's shape exactly (same field names,
  same nesting).
