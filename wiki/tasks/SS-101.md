# Repository & skills schema + validated-only query

## Summary

Replace the current minimal `repositorySchema` in `functions/src/services/repositories-store.ts`
(`repositoryPath`, `author`, `commitHash` only) with the full shape from
`wiki/project_description.md`: a `repositories` doc (owner, repo reference, commit hash,
README-derived description, calculated validation status, calculated install count,
timestamps) and a `skills` subcollection (name, validation status, `findings` array —
`{ severity: 'critical' | 'recommendation', message: string }[]` — install count,
timestamps). Add a query function that returns only docs whose calculated status is
`validated`, for the public catalog ([FR-UI1](../architecture.md#fr-ui1)). This is the single schema every other
story's backend task reads/writes through (AD-1, AD-2). This is also where the
per-field aggregation helpers live: `recalculateInstallCount(repoId)` (AD-11, used by
SS-504) and `recalculateValidationStatus(repoId)` (AD-13, used by SS-401) — each
recomputes exactly one calculated field from the current `skills` subcollection, never
both at once.

## Unit test requirements

- The repository schema accepts a fully-populated payload and rejects one missing a
  required field or with a wrong-typed field.
- The skill schema accepts a payload with an empty `findings` array (not yet validated)
  and one with populated findings, and rejects a finding missing `severity` or `message`.
- Server-side timestamp assignment overrides any client-supplied `created`/`updated`.
- The validated-only query filter excludes `pending` and `failed` documents.

## Integration test requirements

- Writing a repository + skills subcollection via the store against the Firestore
  emulator persists every field and no raw skill file content.
- Listing against the emulator with the public filter returns only
  calculated-status = `validated` documents.
