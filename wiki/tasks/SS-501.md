# Repository document schema

## Summary

Define the Firestore repository document in the single top-level collection: owner
reference, repository reference, GitHub commit hash, short README-derived description,
calculated repo-level validation status, calculated installation count, and
created/updated timestamps. Skill files are never stored — only the commit hash.

## Unit test requirements

- The repository document schema accepts a payload containing owner, repository
  reference, commit hash, description, calculated validation status, calculated
  install count, and timestamps
- The repository document schema rejects a payload missing any required field or with
  a field of the wrong type
- Server-side timestamp assignment logic overrides any client-supplied
  `created`/`updated` values before a document is persisted

## Integration test requirements

- Writing a new repository document via the Cloud Function against the Firestore
  emulator persists all required fields and stores no raw skill file content
- Attempting to write a repository document that is missing required fields, or has
  wrong types, is rejected against the Firestore emulator and no partial document is
  persisted
- Writing a repository document with client-supplied `created`/`updated` timestamps
  results in the server-managed timestamps being stored instead, verified against the
  Firestore emulator
