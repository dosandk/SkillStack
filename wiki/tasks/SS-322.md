# Commit-hash update detection

## Summary

Compare the repository's latest commit hash against the hash stored in Firestore to
detect whether the repository owner has updated it since it was last recorded. Based on
the comparison and the stored version's validation status, decide whether to proceed
with the stored data, install the latest version directly, or defer to the
version-choice flow (SS-323).

## Unit test requirements

- Latest commit hash equals the stored hash: comparison reports no change, and the
  decision logic proceeds without any update handling.
- Latest commit hash differs and the stored version is unvalidated: comparison reports
  a change, and the decision logic resolves to "install latest directly" without
  routing to the version-choice flow.
- Latest commit hash differs and the stored version is validated: comparison reports a
  change, and the decision logic resolves to "defer to version-choice flow" without
  auto-installing.
- Malformed or missing stored commit hash is handled without throwing (treated as a
  change).

## Integration test requirements

- Run the CLI against a fixture repo whose latest commit hash matches the stored
  hash (via a mocked registry lookup) and assert it installs using the stored info
  as-is, with no update prompt or warning shown.
- Run the CLI against a fixture repo whose latest commit hash differs from an
  unvalidated stored version and assert it installs the latest version directly, the
  same as a first-time install, and that the stored commit hash is updated afterward.
- Run the CLI against a fixture repo whose latest commit hash differs from a validated
  stored version and assert the CLI does not auto-install, instead invoking the
  version-choice prompt (SS-323).
