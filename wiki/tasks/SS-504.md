# Firestore security rules for repos & skills

## Summary

Author Firestore security rules so public reads are limited to validated content, owners
can manage their own repositories/skills, and derived fields (validation status, install
counts) are writable only by trusted functions.

## Unit test requirements

- Helper predicate functions embedded in the rules (e.g. `isOwner()`,
  `isDerivedField()`) return the correct boolean result for representative
  allow/deny inputs when evaluated in isolation
- The security rules file passes syntax/schema validation with no errors

## Integration test requirements

- An authenticated owner can write to their own repository/skill document within
  allowed fields against the Firestore emulator, and the write succeeds
- A non-owner authenticated user attempting to write another user's
  repository/skill document is denied by the rules against the Firestore emulator,
  and the data remains unchanged
- Any client (owner or not) attempting to directly write calculated validation
  status or install count fields is denied by the rules against the Firestore
  emulator; only a trusted-function write succeeds, and public reads remain
  limited to validated content
