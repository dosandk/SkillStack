# Create pending-status registry entry

## Summary

Implement the create-entry Cloud Function that SS-411's upload form calls. Given a
repository/skill URL and the caller's identity, the function checks Firestore for an
existing entry for that repo: if none exists, it creates a repository document (plus its
skills subcollection) with `pending` validation status and records the calling user as
owner; if an entry already exists, it reuses/updates that entry instead of creating a
duplicate. The function returns the resulting entry (new or existing) so the client can
render it as pending.

This function requires an authenticated caller and writes an `owner` field — unlike
`SS-512` (the CLI's anonymous post-install telemetry function, which upserts the same
kind of repository/skills document but writes install counts instead of an owner). The
two are separate Cloud Functions with different auth requirements, but should share the
same underlying "upsert repository/skills document, defaulting new ones to `pending`"
helper rather than each reimplementing that create-vs-reuse logic independently.

## Unit test requirements

- Dedup-decision logic: given an existing-repo lookup result, correctly decides
  create-new vs. reuse-existing for both the "not found" and "found" cases
- Pending-status assignment: a newly created repository document (and its skills
  subcollection) is built with `pending` validation status
- Owner-field assignment: the calling user's identity is attached to a newly created
  entry, and is left unchanged when reusing an existing entry

## Integration test requirements

- Calling the function against the Firestore emulator with a repository URL never seen
  before creates a repository document plus its skills subcollection, both with `pending`
  status
- Calling the function again with the same URL does not create a second repository
  document; the existing document is reused/updated and no duplicate skills subcollection
  is created
- Calling the function as an authenticated user records that user's ID as owner on the
  written repository document
