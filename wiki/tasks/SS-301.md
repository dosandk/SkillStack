# `scanRepository` Cloud Function

## Summary

Implement the skill-discovery algorithm (AD-6) as its own Cloud Function: given a
GitHub repo URL, walk its tree via the GitHub API, match `SKILL.md` up to nesting depth
3, and take the whole matched directory regardless of its own depth (rule per
`wiki/project_description.md`). Plain deterministic code — no LLM involved. In the same
call, check Firestore (via SS-101's store) for whether the repository already exists and
its stored commit hash/validation status. Returns metadata only — skill list/paths,
commit hash, validation status, README blurb — never file content (AD-6). This function
is shared: both the CLI (story-cli-install-new) and this story's upload UI call it.

## Unit test requirements

- Given a mocked GitHub tree response, the discovery function returns every path whose
  final segment is `SKILL.md` at depth ≤ 3, and none deeper.
- Once a `SKILL.md` match is found, the returned skill entry's directory includes all
  of that directory's contents regardless of their own nesting depth.
- A repo with no `SKILL.md` within 3 levels returns an empty skill list, not an error.
- The Firestore-existence check returns `{ known: false }` for an unseen repo and
  `{ known: true, commitHash, statuses }` for one already stored.

## Integration test requirements

- Calling the function against a stubbed GitHub API and the Firestore emulator returns
  the expected skill list plus the correct known/unknown-repo branch.
